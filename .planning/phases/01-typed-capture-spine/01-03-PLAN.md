<objective>
Add the **durable job-queue seam** so feature-extraction / separation work is enqueued on capture, with retry and backpressure semantics — at solo scale this is a Postgres-backed queue (sqlxmq), not NATS/Redis (CAP-07, CONTEXT Job Queue decision, STACK.md seam decision §5).

Phase 1 ENQUEUES only: the `JobQueue` trait + typed `JobEnvelope` + an **in-memory** impl that exercises retry/backpressure/durability semantics in RAM-safe unit tests, plus wiring so `capture` enqueues a `FeatureExtract` job. The real consumers (the feature worker) are Phase 2; the production sqlxmq impl is Plan 04 (env-gated). A trivial in-test consumer demonstrates retry/backpressure here without a DB.

Purpose: prove the queue contract (durability, retry, backpressure) on the 4GB machine before the heavy sqlxmq/Postgres leaf exists.
Output: `job.rs` (trait + envelope + retry policy), `queue_mem.rs` (in-memory impl + tests), capture wired to enqueue.
</objective>

  <behavior>
    - enqueue→consume is FIFO; consume marks InProgress and returns None when empty.
    - mark_retry increments attempts; at attempt == max_attempts the record becomes DeadLettered and is not redelivered.
    - enqueue beyond `capacity` returns `Err(JobError::Full)` (backpressure).
    - "durability within process": records persist in the queue across many enqueue/consume cycles until explicitly mark_done/dead-lettered (models the at-least-once contract sqlxmq provides across restart — the cross-restart proof is Plan 04, env-gated).
  </behavior>
  <action>

Add a trivial in-test consumer loop that drains and, for a deliberately-failing job, calls `mark_retry` until dead-letter — asserting the attempt ceiling bounds the retries (no infinite loop). Register the module in `adapters/lib.rs`. Default features only (no tokio).
  </action>
