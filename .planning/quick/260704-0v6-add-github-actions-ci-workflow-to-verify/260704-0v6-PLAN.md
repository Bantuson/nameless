Create `.github/workflows/ci.yml`. Name the workflow `CI`. No `.github/` directory exists yet — this is the first workflow.

**Triggers:** `push` to `main`, `pull_request` to `main`, and `workflow_dispatch`.

**Top level:** set `permissions: contents: read` (no secrets used anywhere; PR-safe). Add a `concurrency` group keyed on `ci-${{ github.ref }}` with `cancel-in-progress: true`. All jobs run on `ubuntu-latest`. Define four sibling jobs with NO inter-job dependency declarations — each job stands alone so a Rust compile failure cannot mask Python/web results (this repo has never been compiled; independent failure surfaces are the point).

Include a short header comment in the YAML explaining that this workflow is the verification vehicle for the env-gated work in `.planning/STATE.md` (repo built on a 4GB no-toolchain machine; CI is the first real compile).

**Job 1 — `rust-lean`** (`timeout-minutes: 30`):
- `actions/checkout@v4`
- `dtolnay/rust-toolchain@stable`
- `Swatinem/rust-cache@v2` (auto-keys per job; no config needed)
- `cargo build --workspace`
- `cargo test --workspace`
- CRITICAL: do NOT pass `--locked` — no `Cargo.lock` exists in the repo (never compiled); the first build generates it. Do not add a step that commits the lockfile, and do not mention this flag in YAML comments (Task 2's gate scans the file for it).

**Job 2 — `rust-postgres`** (`timeout-minutes: 45`):
- `services.postgres`: image `pgvector/pgvector:pg16` (ships the pgvector extension that `migrations/0001_init.sql` enables via `create extension if not exists vector`). Env: `POSTGRES_USER: nameless`, `POSTGRES_PASSWORD: nameless`, `POSTGRES_DB: nameless`. Map port `5432:5432`. Health options: `--health-cmd pg_isready --health-interval 10s --health-timeout 5s --health-retries 5`.
- Job-level env: `DATABASE_URL: postgres://nameless:nameless@localhost:5432/nameless`. This MUST be job-level (not step-level) so it is present both when migrations run and at `cargo build` time — sqlx `query!`/`query_as!` macros connect to the database during compilation, and there is no `.sqlx` offline cache in this repo.
- Steps: checkout, `dtolnay/rust-toolchain@stable`, `Swatinem/rust-cache@v2`, then a migration step BEFORE any cargo invocation: `for f in migrations/*.sql; do psql "$DATABASE_URL" -v ON_ERROR_STOP=1 -f "$f"; done` — plain `psql` is preinstalled on ubuntu-latest; the shell glob applies 0001→0004 in lexicographic order; `ON_ERROR_STOP=1` makes a failed migration fail the job instead of silently continuing.
- Build + test with the heavy feature. CRITICAL cargo detail: this is a VIRTUAL workspace (root `Cargo.toml` has no `[package]`), so a bare root-level feature flag is rejected by cargo. Use member/feature syntax: `cargo build --workspace --features nameless-adapters/postgres,nameless-cli/postgres,nameless-api/postgres` and the same flags for `cargo test --workspace`. `nameless-core` has no such feature — the pkg/feature syntax is what makes this work.
- Add a YAML comment on this job noting it is the job that verifies review finding P7 CR-01 (nullable PG columns vs non-`Option` fields in `get_context_summary` — only compiles if the fix holds under real sqlx verification).

**Job 3 — `python`** (`timeout-minutes: 15`):
- `strategy.fail-fast: false` with `matrix.project: [workers, knowledge-pipeline]` so one project's failure doesn't cancel the other.
- `defaults.run.working-directory: ${{ matrix.project }}`.
- Steps: checkout, `astral-sh/setup-uv@v6` with `enable-cache: true` and `cache-dependency-glob: "${{ matrix.project }}/pyproject.toml"`, then `uv sync --python 3.12 --extra dev` (both projects declare `requires-python >=3.11,<3.13`; 3.12 is the safe pin), then `uv run pytest`.
- Do NOT install the heavy extras (`ml`, `pg`, `ingest`, `asr`, `extract`, `embed`) — both test suites run entirely against fakes on the light base per the pyproject tier comments; pulling torch/librosa in CI would be slow, wasteful, and unnecessary.

**Job 4 — `web`** (`timeout-minutes: 15`):
- `defaults.run.working-directory: web`.
- Steps: checkout, `actions/setup-node@v4` with `node-version: 22`, `cache: npm`, `cache-dependency-path: web/package-lock.json` (lockfile exists — `npm ci` is valid), then `npm ci`, `npm test` (script is `vitest run` — non-watch, CI-safe), and `npm run build` (script is `tsc --noEmit && vite build`, so this single step covers both the typecheck and the production build; no separate `npm run typecheck` step needed).

**Action pinning:** pin all actions to major tags exactly as listed above (`@v4`, `@v2`, `@v6`, `@stable`). Do not use `@master`/`@main` refs and do not introduce any action beyond the five named.
