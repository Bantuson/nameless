<objective>
Add a no-API synthesis path to the knowledge-pipeline, mirroring the just-shipped FileClaimExtractor
(commits 6bd7c20..4dd9a4f): a `FileSkillSynthesizer` adapter that reads one pre-drafted skill JSON per
production cell from a directory, plus a `skills synthesize --drafts-dir <dir>` CLI plane that wires it to
the REAL FilesystemCorpusStore + REAL SqliteClaimStore + REAL FilesystemSkillStore through the UNCHANGED
`SynthesisPipeline`.

Purpose: the user has no Anthropic API credits. Claude Code (in-session) authors skill drafts file-to-file
as `emit_skill` tool payloads; the pipeline then ingests them so the pure `citation_gate` (R1–R5) judges
them exactly like API output — a draft with an invented number or an uncited claim is REJECTED identically.
The gate stays identical — only the synthesizer leaf changes.

Output: new adapter module + eager export, one new CLI plane + `--drafts-dir` flag, one new test file.
`SynthesisPipeline`, `citation_gate`, and `layered_emitter` are NOT modified.
</objective>

<design_notes>
Load-bearing facts discovered during planning — do not re-derive, do not deviate:

1. **There is NO per-cell error seam in SynthesisPipeline** (unlike MiningPipeline, which try/excepts
   per-video — the seam the FileClaimExtractor plan leaned on). `SynthesisPipeline.synthesize` calls
   `self._synth.synthesize(cell, cell_clusters)` bare: any exception aborts the entire run. Therefore
   "missing draft file → skip" CANNOT be implemented by raising from the adapter. It is implemented by
   **scoping cell selection**: the pipeline derives its cells from `claim_store.list_clusters()` via the
   pure `select_cells` (a cell exists iff some cluster's `genre` list evidences it, cell_selection.py).
   The drafts plane wraps the REAL `SqliteClaimStore` in a thin read-only view (`CellScopedClaimStore`)
   whose `list_clusters()` trims each cluster's `genre` list to genres whose `(stage, genre)` cell has a
   draft file (dropping clusters left with no genre). Missing-file cells are then never selected — a
   natural skip — and the CLI prints one clear SKIP line per missing cell (to stderr, so `--json` stdout
   stays parseable). This satisfies "flow through the UNCHANGED SynthesisPipeline" with zero pipeline edits.
2. **Scoping is output-invariant for the cells that DO run.** The pipeline calls exactly two methods on
   the claim store: `list_clusters()` and `list_claims()`. The view passes `list_claims()` through
   UNFILTERED (the gate's authoritative claim index stays complete) and everything else delegates via
   `__getattr__`. `clusters_for_cell` membership for an available cell is unchanged (its genre is kept in
   every cluster that had it), the claims inside clusters are untouched, and R5 snapshots come from the
   real corpus — so an available cell's authored skill is byte-identical to an unscoped run. Cluster
   models are frozen pydantic; trim genre lists with `model_copy(update={"genre": kept})`.
3. **Malformed file → loud ABORT (differs from the mining plane, deliberately).** With no per-cell seam
   and the pipeline off-limits, a malformed present file cannot become a per-cell skip. It is also not a
   selection problem — the file exists, its CONTENT is broken, which is a fatal authoring error the user
   must fix. The adapter raises `DraftFileError(ValueError)` naming the file; `_handle_synthesize`
   converts it (drafts mode only) to `SystemExit` for a clean, loud, traceback-free exit.
4. **Parse-failure must raise, never template-fallback.** `AnthropicSkillSynthesizer` falls back to
   `template_synthesize` when `parse_synthesizer_output` returns `None`. The file adapter MUST NOT — that
   would author content the human never drafted. `None` (pydantic-invalid payload, default with no valid
   claim ids after re-grounding, or blank default body) → `DraftFileError` naming the file and the reason.
   Distinguish this **structural** failure from a **content** failure: a draft that parses but asserts an
   invented number / uncited prose flows to the UNCHANGED `citation_gate` and shows as `REJECTED` with
   reasons in the report — exactly the behavior demanded for API output.
5. **Payload shape.** The JSON file body is exactly the `emit_skill` tool input described by
   `EMIT_SKILL_TOOL_SCHEMA`: `{name, description, default: {body, claim_ids, stance?}, sections:
   [{kind, topic, technique, stance?, body, claim_ids}]}`. Citations are id-references only —
   `parse_synthesizer_output(raw, cell, clusters, prompt_version=...)` re-grounds every citation's
   quote/timestamp/source from the REAL claims and drops ids outside the cell, so a file can never
   fabricate a receipt. Reuse it verbatim; invent nothing.
6. **File naming (collision-safe, documented):** `{normalize_key(stage)}__{normalize_key(genre)}.json`,
   e.g. `vocal-layering__rnb.json`, `drums__amapiano.json`, `bassline__deep-house.json`. `normalize_key`
   emits only `[a-z0-9-]`, so the `__` separator can never occur inside a key — the split is unambiguous
   even though stages AND genres contain hyphens (`deep-house`, `alt-piano`, `vocal-layering` make any
   single-hyphen or slug-based scheme ambiguous: the cell slug `deep-house-drums` has two readings).
   Path safety is by construction: filenames are composed ONLY from `normalize_key` output (cluster-derived
   stage/genre labels), never from raw user input, and `[a-z0-9-]` admits no separators or dot-dots.
7. **Provenance:** module constant `FILE_DRAFT_PROMPT_VERSION = "file-draft-v1"` passed as
   `prompt_version` so `skills show`/audit reveal a file-drafted skill's origin.
8. **The 5 fixture P1 cells** (from `mine_fixture_claim_layer` in tests/conftest.py, proven by
   test_synthesis_pipeline): `rnb-vocal-layering`, `amapiano-drums`, `amapiano-bassline`,
   `deep-house-bassline`, `rnb-bassline`. Tests should derive cells dynamically via
   `select_cells(claim_store.list_clusters())` rather than hardcoding, but these are the expected set.
9. **This is directly runnable here**: tests use fixtures/tmpdir/sqlite only. Verify steps MUST actually
   run `uv run pytest` (from `knowledge-pipeline/`).
</design_notes>
